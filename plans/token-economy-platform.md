# Token 經濟平台 — 計劃書 v0.1

> 一個以 Token 為核心、但**對一般消費者隱藏複雜度**的多用途平台：
> 支付、生成內容、內容販售、Agent-to-Agent 交易、閒置 API/算力租賃，
> 透過 **Stripe（法幣 + SPT/ACP）** 與 **x402（agent 穩定幣 rail）** 雙軌結算，
> 並以**電商 + 實體錨定商品（龍蝦／生鮮）**作為信任橋樑，形成「生產內容 → 賺 Token → 再消費」的循環飛輪。

狀態：草稿 · 最後更新 2026-06-09 · 分支 `claude/token-economy-platform-43k476`

---

## 0. TL;DR — 一般消費者要從哪個方向切入？

**核心結論：不要從 Token 切入。從「一個他們本來就想買的真東西」切入。**

一般消費者對「token 經濟」三個字是反感或無感的（聯想到投機、詐騙、看不懂的錢包）。
所以 MVP 的對外敘事應該是：**「一個會幫你生內容、還能買到好東西（龍蝦/生鮮/數位商品）的平台」**，
Token 在第一階段是**隱形的後台記帳單位 + 回饋點數**，不是賣點。

切入順序（由易到難、由信任到投機）：

1. **法幣優先**：所有人用 Stripe 信用卡/Apple Pay 就能完成第一筆交易，零錢包、零 gas、零術語。
2. **點數回饋**：消費與「生產內容」會賺到 Token，但對外只說「點數/Credits」，可折抵下一筆。
3. **效用解鎖**：Token 累積到一定量，解鎖「生成內容」「販售內容賺更多」等進階功能 → 用戶自然理解 Token 有用。
4. **Agent / 租賃 rail**：只有開發者與重度用戶才會碰到 x402 / A2A / API 租賃，這是 B 端與進階層，不要塞給一般消費者。

一句話：**「先賣真東西，Token 最後才出現。」** 用實體商品的確定性，去背書 Token 的不確定性。

## 1. 產品定位與一句話

- **對一般消費者**：能生成內容、又能買到優質商品的會員制電商 + 創作平台，消費與創作都回饋點數。
- **對創作者**：把你產出的內容直接掛上架販售，賺取可折抵、可提現的 Token。
- **對開發者 / Agent**：把你閒置的 API 額度、算力、資料服務掛上 x402 paywall，讓別人（或別人的 AI agent）按次付費租用。

三種人，同一個 Token，三種速度的信任曲線。

## 2. 問題診斷：為什麼一般消費者不接受 Token 經濟

| 障礙 | 一般消費者的真實感受 | 本計劃的對策 |
|---|---|---|
| 錢包/助記詞 | 「我會弄丟、會被盜」 | 第一階段不需要錢包，托管式帳戶 + Stripe |
| Gas / 鏈上術語 | 「為什麼買個東西要付手續費還要等」 | x402/穩定幣在後台，前台只有「點數」與「台幣/美元」 |
| 波動 | 「今天 100 明天 50」 | Token 以**穩定幣（USDC）**或**法幣計價的點數**錨定，避免投機定價 |
| 投機/詐騙觀感 | 「這是不是又一個割韭菜」 | 用實體商品（龍蝦/生鮮）+ Stripe 退款 + 可隨時換回法幣建立信任 |
| 沒有理由用 Token | 「我為什麼不直接刷卡」 | Token 唯一不可取代的場景：**創作變現 + Agent 自動交易 + 租賃** |

**設計原則**：法幣是「入口與退出」，Token 是「內部超能力」。讓用戶**因為功能而留下，而不是因為投機而進來。**

## 3. Token 設計（效用優先，避免證券化）

### 3.1 角色：單一 Token、四種用途
1. **支付媒介** — 站內購買商品、內容、服務。
2. **創作報酬** — 生產並售出內容，賺取 Token。
3. **A2A 結算單位** — agent 之間互相呼叫服務時的計價與結算。
4. **租賃計費單位** — 閒置 API / 算力 / 額度按次計費。

### 3.2 定價錨定（關鍵）
- Token **不做浮動市場定價**，而是**以 1 Token = 固定法幣/USDC 價值**（如 1 Token = $0.01）發行與回收。
- 這讓它在法律上更接近**預付點數 / 穩定幣 voucher**，而非投機性證券資產，降低監管與消費者疑慮。
- 後台真正在鏈上流動的是 **USDC（透過 x402）**；站內 Token 是其上的記帳層（可隨時 1:1 兌付）。

### 3.3 合規護欄（必須在計劃書內就寫清楚，否則消費者與支付商不會接受）
- **不承諾增值、不做二級市場炒作**，定位為效用點數而非投資品。
- **預付價值監管**：可能落入電子票證 / 儲值 / money transmitter 範疇 → 早期用 Stripe + 托管帳戶規避自行持有資金的責任。
- **KYC/AML**：提現與大額租賃收入觸發 KYC；小額消費維持低摩擦。
- **稅務**：創作者收入與租賃收入需可開立憑證/報表。

> 待確認（見 §11）：發行地法域、是否真的需要上鏈、是否做提現（提現 = 監管難度跳一級）。

## 4. 雙軌支付架構：Stripe vs x402，何時用哪個

這是整個專案的技術心臟。**兩條 rail 並存，按「誰在付錢」分流。**

### Rail A — Stripe（人 → 平台，法幣世界）
- **用途**：一般消費者刷卡買商品/內容、儲值點數、訂閱。
- **武器**：
  - 標準 Checkout / Payment Intents（法幣入口）。
  - **ACP（Agentic Commerce Protocol，Stripe×OpenAI）**：當用戶透過 ChatGPT 等 AI 入口下單時的標準結帳流。
  - **SPT（Shared Payment Token）**：讓 AI agent 用「用戶已存的付款方式」發起付款，**不暴露卡號**——這正是你說的「SPT」。
- **角色**：信任入口 + 退款 + 開發票 + 合規收單。**一般消費者 100% 走這條。**

### Rail B — x402（agent → agent / 服務按次付費，穩定幣世界）
- **用途**：A2A 交易、API/算力租賃、內容/服務的程式化按次付費（micropayment）。
- **武器**：
  - HTTP `402 Payment Required` + USDC，**無帳號、無 session、無 gas 摩擦給呼叫方**。
  - 零協議費、已在 Base / Solana 上線、2026 已是 Linux Foundation 標準。
  - 可掛 **AP2（Google Agent Payments Protocol）** 做 agent 身份、授權、政策與不可竄改收據。
- **角色**：機器經濟的計費骨幹。**開發者與 agent 走這條；一般消費者看不到。**

### 分流規則（一句話）
> **是人、用法幣、要退款 → Stripe。是機器、按次付、要自動化 → x402。Token 是兩條 rail 之間的共同記帳單位。**

### 協議速查（2026 現況）
| 協議 | 主導者 | 適合 | 在本專案的位置 |
|---|---|---|---|
| **ACP** | Stripe + OpenAI | AI 入口的人類結帳 | Stripe rail 的標準結帳 |
| **SPT** | Stripe | agent 用已存付款方式代付 | Stripe rail 的 agent 代付 |
| **AP2** | Google (A2A 之上) | agent 身份/授權/收據 | x402 rail 的信任層 |
| **x402** | Coinbase / LF | 穩定幣按次微支付 | 機器經濟結算骨幹 |
| **MPP** | （比較生態之一） | 評估中 | 候選，暫不採用 |

## 5. 產品模組

1. **電商 / 實體商品（信任錨）** — 龍蝦、生鮮或精選實體/數位商品。**確定性高、消費者熟悉**，作為拉新與信任建立的第一步。Stripe 收單、可退款。
2. **內容生成** — 平台提供 AI 生成（文字/圖/影音）。消費 Token 來生成；生成成本由後台 API（你的或租來的）支撐。
3. **內容販售市集** — 用戶把生成或原創內容上架販售，**售出即賺 Token**。這是「生產→變現」的關鍵節點。
4. **A2A 交易** — 創作者/開發者把服務包成 agent，掛 x402 paywall，供其他 agent 自動呼叫付費。
5. **閒置資源租賃** — 把多餘的 API 額度 / 算力 / 訂閱權益按次租給別人或別人的 agent，x402 計費。

## 6. 循環飛輪（Token 的存在理由）

```
        ┌──────────────────────────────────────────────┐
        │                                              │
   買實體商品/儲值 ──(Stripe 法幣)──► 獲得 Token 點數    │
        │                                              │
        ▼                                              │
   用 Token 生成內容 ──► 上架販售 ──► 售出賺回 Token ───┘
        │                                   ▲
        ▼                                   │
   閒置 API/算力 掛 x402 ──► 被 agent 租用 ──┘（賺 USDC→Token）
```

- **入口**：法幣（低摩擦、人人會）。
- **黏著**：生成 + 變現（給人「不只是花錢，還能賺」的理由）。
- **規模**：A2A + 租賃（機器交易量遠大於人，是天花板）。
- **退出**：Token 可 1:1 折抵或（KYC 後）提現 → 降低「被困住」的恐懼。

## 7. 分階段路線圖

### Phase 0 — MVP（0–3 月）：「一個會賺點數的電商」
- Stripe Checkout + 實體商品（龍蝦/生鮮）上架。
- Token 以「Credits 點數」呈現，消費回饋、可折抵。**完全不提 crypto/錢包。**
- 內容生成 v1（消耗點數）。
- 目標：證明有人願意付錢、回購、並用點數生成內容。

### Phase 1 — 創作變現（3–6 月）：「生產內容賺點數」
- 內容販售市集上線，售出賺 Token。
- 後台導入 USDC 結算（仍對用戶隱形），托管帳戶。
- 開放有限度提現（觸發 KYC）。

### Phase 2 — 機器經濟（6–12 月）：「Agent rail」
- x402 paywall + API/算力租賃 marketplace。
- AP2 / A2A 整合，agent 自動互相付費。
- SPT/ACP：讓外部 AI 入口（ChatGPT 等）能直接在你平台結帳。

### Phase 3 — 規模化
- 開放第三方商品與服務、開發者生態、Token 作為跨服務通用結算單位。

## 8. 信任與消費者體驗（讓「一般人」真的敢用）

- **零錢包啟動**：email 註冊即可，托管帳戶，私鑰用戶不用碰。
- **永遠看得到法幣價**：所有 Token 旁顯示等值台幣/美元。
- **可退款**：實體商品走 Stripe 標準退款流程 → 與一般網購無異。
- **可退出**：Token 可折抵或提現，不做「只進不出」的封閉點數。
- **實體背書**：收到真的龍蝦/商品 = 對平台信任的實體證明，遠勝任何白皮書。
- **客服與爭議**：人類客服 + 明確爭議流程（agent 經濟尤其需要可仲裁的收據，AP2 提供）。

## 9. 風險

| 風險 | 說明 | 緩解 |
|---|---|---|
| 監管（儲值/證券/匯兌） | 自持資金與提現是高危區 | Stripe + 托管、效用定位、法律意見、分法域上線 |
| 消費者不買單 Token | 講太多 crypto 會嚇跑人 | Phase 0 完全隱藏 Token，先賣實體 |
| 雙邊冷啟動 | 創作者與買家、租方與用方互為前提 | 用實體電商先做單邊現金流，再長出市集 |
| 內容品質/濫用 | AI 內容氾濫、版權、詐騙 | 審核、版權檢查、信譽分、x402 收據可追溯 |
| 技術依賴 | x402/Stripe/協議仍在演進 | 抽象出支付介面層，rail 可替換 |
| 龍蝦/生鮮履約 | 冷鏈、損耗、退貨 | 先選高毛利、易履約的 SKU 驗證，再擴品類 |

## 10. 關鍵指標（KPI）

- Phase 0：首購轉換率、回購率、點數折抵使用率、內容生成次數。
- Phase 1：上架創作者數、內容售出 GMV、創作者賺取→再消費的「循環率」。
- Phase 2：x402 交易筆數、API/算力租賃供需數、agent 自動交易佔比。
- 全程：Token 流通速度（velocity）、提現/折抵比、CAC vs LTV。

## 11. 待你確認的關鍵決策

1. **「龍蝦」** = 真的做生鮮/海鮮電商作為實體錨，還是它是其他品類/代稱？（本計劃暫以「高慾望實體商品」解讀，可替換。）
2. **發行/營運法域**：台灣？美國？這直接決定儲值與提現的監管路徑。
3. **Token 是否真的上鏈**，還是先做「站內點數 + 後台 USDC」即可？（建議後者起步。）
4. **是否提供提現**（換回法幣）？提現會讓監管難度跳一級，但對信任很關鍵。
5. **目標首批用戶**：先打 C 端消費者，還是先打有閒置 API/算力的開發者？（本計劃建議 C 端實體電商先行。）
