通用大型語言模型強，但在**特定領域做深度分析**時有兩個硬傷：知識時效性落後，以及對小眾領域的事實稀釋。以 Polkadot 生態為例，2.0 版本的 Agile Coretime、RegionX、JAM 這些 2024–2026 之間的關鍵迭代，通用模型要嘛答不出、要嘛抓到過時版本。

解法不是換更大的模型 — 是**給模型一個結構化、可信的外部知識庫**，然後在前面包一層人格與任務 prompt。這就是我在做 myGPTs 的動機。

## 背後的系統：Obsidian Wiki + LLM Ingest Loop

這兩個 GPT 都不是瞎調出來的。它們共用一個 Obsidian vault 當知識骨架：

```
wiki/
├── sources/      一源一檔，逐篇 ingest 的原始資料
├── entities/     人物、公司、產品的實體頁
├── concepts/     協定、架構、產業鏈等概念
└── synthesis/    跨源整合後產出的對外長文
```

每次讀到新資料（podcast、論文、財報），我會走這條流程：

1. 和 Claude Code 討論 key takeaway
2. 寫成 `sources/<slug>.md`
3. 更新或新開相關的 `entities/` 與 `concepts/`（一次 ingest 通常牽動 5–15 頁）
4. 更新 `index.md` 與 `log.md`

這個 vault 的**文字內容**就是餵給 Jamia/Spacesharks 的素材。當讀者問一個跨源的問題，模型其實是在我手工建的 context graph 上做檢索，不是在整個 internet 裡面猜。

## 為什麼不直接用 RAG 或 API 串？

我試過。對一個人維護的知識庫來說，Custom GPT 的優勢是：

| 需求 | Custom GPT | 自建 RAG |
|------|-----------|---------|
| 零後端、免費分享 | ✅ | ❌ 要 host vector store |
| 10 MB 以下的知識體量 | ✅ 直接上傳 | ⚠️ 過度工程 |
| 前端 UX（手機也能聊） | ✅ ChatGPT app | ❌ 要自己寫 |
| 客製工具（抓新聞 / 查價） | ⚠️ action 有限 | ✅ 完全可控 |

等知識規模或互動模式突破 Custom GPT 的上限，再搬去 RAG / Agent 架構也不遲。**先把知識寫對、再談系統**。
